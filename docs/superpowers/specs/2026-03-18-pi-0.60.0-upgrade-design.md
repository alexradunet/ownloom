# Pi 0.60.0 Upgrade Design

**Date:** 2026-03-18
**Status:** Draft

## Overview

Upgrade pi from 0.57.1 to 0.60.0, replace the `llm-agents-nix` flake dependency with a direct `buildNpmPackage` derivation, integrate `createLocalBashOperations()` into `bloom-audit` for automatic bash command logging, and acknowledge the startup auto-update breaking change.

## Scope

| Change | Files |
|--------|-------|
| Drop `llm-agents-nix`, add `core/os/pkgs/pi/default.nix` | `flake.nix`, new derivation |
| Bump pi to 0.60.0 | `package.json`, `core/os/pkgs/pi/default.nix` |
| Recompute bloom-app npm deps hash | `core/os/pkgs/bloom-app/default.nix` |
| Remove offline llm-agents-nix ISO embedding | `core/os/hosts/x86_64-installer.nix`, `flake.nix` specialArgs |
| Bash audit logging via `createLocalBashOperations()` | `core/pi-extensions/bloom-audit/index.ts` |

Out of scope: `--fork` daemon session bootstrapping (deferred).

## 1. Pi Nix Derivation

### Problem

Pi is currently delivered via `llm-agents-nix` (`github:numtide/llm-agents.nix`), an external flake maintained by a third party. This adds a flake input to track, an indirect update path, and potential build latency from the external rebuild chain.

### Design

A new derivation at `core/os/pkgs/pi/default.nix` uses `buildNpmPackage` to build `@mariozechner/pi-coding-agent` directly from the npm registry tarball. This follows the same pattern as other custom derivations in `core/os/pkgs/`.

```
core/os/pkgs/pi/default.nix   ŌåÉ new file
```

In `flake.nix`:
- Remove `llm-agents-nix` input
- Replace `piAgent = llm-agents-nix.packages.${system}.pi` with `piAgent = pkgs.callPackage ./core/os/pkgs/pi {}`
- Remove `llmAgentsSrc` from the `specialArgs` block passed to the ISO target

All downstream consumers of `piAgent` (`bloom-app/default.nix` node_modules symlinking, `bloom-app.nix` system packages, daemon PATH) are unchanged.

**Update procedure:** bump `version` and `npmDepsHash` in `core/os/pkgs/pi/default.nix`. No external flake to chase.

`package.json` devDependencies (`@mariozechner/pi-coding-agent` and `@mariozechner/pi-ai`) are bumped to exact `"0.60.0"` (no caret) to stay aligned with the pinned Nix derivation. After updating `package.json` and regenerating `package-lock.json`, the `npmDepsHash` in `core/os/pkgs/bloom-app/default.nix` must be recomputed (`nix build` will report the correct hash on first failure).

### ISO Offline Installation

`core/os/hosts/x86_64-installer.nix` currently embeds `llmAgentsSrc` (the llm-agents-nix flake source tree) in the squashfs for offline installation. With the flake input removed:

- Remove the `llmAgentsSrc` parameter from the module signature
- Remove the `environment.etc."bloom/offline/llm-agents-nix"` line

Offline installation continues to work: the new `buildNpmPackage` derivation fetches the pi npm tarball as a fixed-output derivation. Because `piAgent` is already in `environment.systemPackages`, its entire transitive store closure ŌĆö including the npm tarball ŌĆö is bundled into the squashfs. No separate embedding is needed.

Also update the stale `boot.kernelParams` comment in `x86_64-installer.nix` (currently mentions "nixpkgs/bloom/piAgent sources" ŌĆö remove the piAgent phrasing since it is no longer an embedded source tree).

**Sequencing constraint:** the `llmAgentsSrc` removal from `x86_64-installer.nix` and `flake.nix` must land in the same commit that introduces `core/os/pkgs/pi/default.nix`. Removing the embedding before the new derivation is wired up would produce an ISO that cannot install offline.

## 2. `createLocalBashOperations()` in bloom-audit

### Problem

Pi executes bash commands on behalf of the user with no audit trail. There is no way to correlate which commands ran, when, or with what outcome ŌĆö a gap for both debugging and accountability.

### Design

`bloom-audit/index.ts` imports `createLocalBashOperations()` from `@mariozechner/pi-coding-agent` and registers the resulting handler as pi's `user_bash` backend. The wrapper logs each command and its outcome to the audit log before passing through to the real executor unchanged.

This keeps the concern centralized: no other extension needs to add logging, and the wrapper is transparent to all other extensions that invoke bash.

```
core/pi-extensions/bloom-audit/index.ts   ŌåÉ updated
```

**API shape (to be confirmed against pi 0.60.0 source before implementation):**

`createLocalBashOperations()` returns a bash operations object. The registration call is expected to follow this pattern:

```ts
import { createLocalBashOperations } from "@mariozechner/pi-coding-agent";

export default function (pi: ExtensionAPI) {
  const bashOps = createLocalBashOperations();
  // wrap execute to inject audit logging
  pi.setBashOperations({
    ...bashOps,
    execute: async (cmd, opts) => {
      appendAudit({ ts: new Date().toISOString(), event: "bash_invoke", toolCallId: opts?.toolCallId, cmd });
      const result = await bashOps.execute(cmd, opts);
      appendAudit({ ts: new Date().toISOString(), event: "bash_result", toolCallId: opts?.toolCallId, exitCode: result.exitCode });
      return result;
    },
  });
  // ... existing hooks and tools unchanged
}
```

The exact method name (`setBashOperations` or equivalent) and the `execute` signature must be verified from the pi 0.60.0 exported types before writing production code.

Behavioral contract:
- Commands are never blocked or modified ŌĆö the wrapper is pass-through
- Log entries mirror the existing `tool_call`/`tool_result` schema, including `toolCallId` for correlation
- Logging failures do not interrupt command execution

## 3. Breaking Change: Startup Auto-updates Removed

Pi 0.60.0 no longer checks or updates unpinned npm packages at startup. Updates require an explicit `pi update` call; interactive mode notifies of available updates in the background.

**Impact on Bloom:** None. Bloom is not managed by pi's package system ŌĆö it is Nix-managed and loaded via the path reference in `~/.pi/agent/settings.json`. Pi's auto-update behavior never applied to Bloom's own package. No code or documentation changes are required.

## Testing

- `nixos-rebuild switch` succeeds with the new derivation; `pi --version` reports 0.60.0
- Bloom extensions load correctly after rebuild
- Bash commands executed through pi appear in the bloom-audit log with correct metadata
- No regression in existing extension behavior
